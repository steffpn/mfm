  <behavior>
    - Cleanup worker registers a BullMQ job scheduler that runs every 30 seconds
    - Each job scans all subdirectories in data/streams/
    - For each segment file with mtime older than 3 minutes, the file is deleted
    - Empty station directories from deleted/inactive stations are removed
    - Worker uses createRedisConnection (not the shared redis instance) for BullMQ
    - startCleanupWorker returns references to queue and worker for graceful shutdown
    - Worker handles missing/inaccessible directories gracefully (no crashes)
  </behavior>
  <action>
    **1. Create `apps/api/src/workers/cleanup.ts`:**

    Use BullMQ v5 job scheduler API (NOT deprecated repeatable jobs).

    ```typescript
    import { Worker, Queue } from 'bullmq';
    import { createRedisConnection } from '../lib/redis.js';
    import fs from 'node:fs/promises';
    import path from 'node:path';
    import pino from 'pino';

    const logger = pino({ name: 'cleanup-worker' });
    const CLEANUP_QUEUE = 'segment-cleanup';
    const DATA_DIR = path.resolve('./data/streams');
    const MAX_AGE_MS = 3 * 60 * 1000; // 3 minutes per user decision
    ```

    Export `startCleanupWorker()`:
    - Create Queue with `createRedisConnection()`
    - Call `queue.upsertJobScheduler('cleanup-scheduler', { every: 30_000 }, { name: 'cleanup-segments', data: {} })` -- per BullMQ v5.16+ API
    - Create Worker that processes cleanup jobs:
      1. Read all subdirectories in DATA_DIR
      2. For each subdirectory, read all files
      3. For each file, stat and check if `Date.now() - stat.mtimeMs > MAX_AGE_MS`
      4. Delete files older than 3 minutes
      5. After file cleanup, check if directory is empty AND the station is not ACTIVE in DB (query Prisma) -- if so, remove the directory (orphan cleanup for deleted stations)
    - Return `{ queue, worker }` for graceful shutdown

    **2. Wire cleanup worker into supervisor entry point:**

    Modify `apps/api/src/services/supervisor/index.ts`:
    - Import `startCleanupWorker` from `../../workers/cleanup.js`
    - Call `const { queue: cleanupQueue, worker: cleanupWorker } = await startCleanupWorker()` after staggered startup
    - In graceful shutdown, add: `await cleanupWorker.close()` and `await cleanupQueue.close()`

    **3. Update Docker Compose:**

    Modify `docker-compose.yml`:
    - Add `ffmpeg` as a system package in the API service (or create a Dockerfile that extends node:20 with FFmpeg 7.x installed)
    - Add volume mount: `./data/streams:/app/data/streams` for the API/supervisor service
    - Set `ulimit` for file descriptors: `ulimits: { nofile: { soft: 65536, hard: 65536 } }` per Pitfall 5
    - If no API service exists in docker-compose.yml yet, add a comment documenting the expected Dockerfile setup for production (FFmpeg + Node.js)

    Since the current docker-compose.yml only has TimescaleDB and Redis, add a new `api` service with a Dockerfile reference. Create `apps/api/Dockerfile` if needed:
    ```dockerfile
    FROM node:20-bookworm-slim
    RUN apt-get update && apt-get install -y ffmpeg && rm -rf /var/lib/apt/lists/*
    WORKDIR /app
    # Copy and install will be handled by production deploy
    ```

    **4. Update `apps/api/package.json` scripts:**
    - Ensure `"supervisor"` script exists: `"supervisor": "tsx src/services/supervisor/index.ts"`
    - Add `"cleanup"` script: `"cleanup": "tsx src/workers/cleanup.ts"` (for standalone cleanup worker if needed)

    **5. Write tests:**

    `apps/api/tests/workers/cleanup.test.ts`:
    - Mock `fs.readdir`, `fs.stat`, `fs.unlink`, `fs.rmdir`

    - Test: empty directory with active station is NOT removed (station may have just started)
    - Test: non-existent DATA_DIR does not crash
    - Test: permission error on single file does not crash worker (continues to next file)
    - Mock Prisma for station status checks
